# 第二阶段手工测试说明

## 准备

1. 执行 `alembic upgrade head`，确认当前版本为 `0005`。
2. 运行开发种子，使用市民、坐席、部门人员、管理员四个独立账号。
3. 浏览器保留开发者工具 Network 面板，确认所有写操作携带最新 `version` 且响应有 `X-Request-ID`。

## A. 分类管理（管理员）

1. 打开“诉求分类”，依次创建 L1、L2、L3；确认编码重复、第四级、停用上级但仍有启用子级均被拒绝。
2. 为 L3 配置默认责任部门、60 分钟受理时限、1440 分钟办理时限。
3. 停用 L3，确认坐席受理下拉不再出现；重新启用后恢复。
4. 在审计日志验证 `create_category`、`update_category`，资源编号与分类 ID 一致。

## B. 优先级确认（市民、坐席）

1. 市民创建工单并选择/表达“重大事件”；确认新工单正式优先级仍为普通，市民参考值单独展示。
2. 坐席点击受理，选择“路灯故障”和“紧急”；确认受理成功后分类路径、确认优先级、确认时间正确。
3. 用接口直接提交非法优先级或非末级分类，确认返回 422。
4. 检查审计日志同时存在 `accept_ticket` 和 `confirm_ticket_triage`。

## C. SLA 与催办（四角色）

1. 按分类 60/1440 分钟、紧急 50% 校验应受理为创建后约 30 分钟，应办结为受理后约 12 小时。
2. 坐席派发至默认部门，部门人员开始处理。
3. 部门人员暂停计时但不填原因，确认表单或后端拒绝；填写原因后确认状态为“计时已暂停”。
4. 等待至少一分钟后恢复，确认应受理/应办结时间同步顺延、累计暂停增加、暂停原因清空。
5. 市民点击催办，确认次数加一；再次催办确认累计。用其他市民账号访问或催办，确认 403。
6. 提交处理结果后尝试催办或暂停，确认后端返回 409。
7. 构造截止时间在 4 小时内和已过期的数据，确认列表“即将超时”“已超时”筛选与详情标识一致。

## D. 看板（管理员）

1. 确认指标包含工单总数、即将超时、已超时、平均受理时长、平均办理时长。
2. 抽取三张工单手算分钟差，和平均值交叉核对。
3. 按部门核对纳入统计数、超时数和 `超时率 = 超时数 / 纳入统计数 × 100%`。
4. 确认停用部门和分类不会删除历史统计。

## E. 回归与兼容

1. 迁移前备份包含 `low/normal/high/urgent` 的旧数据，升级后核对映射、分类为空、历史记录完整。
2. 对旧工单继续执行受理、派发、处理、解决、评价，确认一期闭环不受影响。
3. 验证四角色菜单和越权路径：市民不能进管理页，坐席不能维护分类，部门人员只能看本部门，管理员可看全局。
4. 在 Chromium、Firefox、WebKit 各执行一次关键闭环；在 1280px 与 390px 宽度检查列表横向滚动、弹窗和 SLA 卡片。
