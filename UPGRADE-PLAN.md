# 倾听助手升级实施方案（UPGRADE-PLAN）

> 依据：`PROJECT-AUDIT.md` 与现有 README  
> 目标：升级为可稳定演示、可单机部署交付、适合求职展示的完整工程作品。  
> 边界：可在受控公网或服务器环境上线展示，但不宣称达到真实政务生产系统的高可用、等保和灾备标准。

## 1. 项目真实现状

### 已完成并验证

- React、FastAPI、PostgreSQL、MinIO、Rasa、Action Server、Duckling 七服务真实存在。
- 市民、坐席、部门人员、管理员四角色权限和数据范围有效。
- 已跑通：建单 → 受理 → 派发 → 办理 → 提交结果 → 不满意重办 → 申诉审核 → 再次办理 → 市民确认 → 办结。
- 状态机、幂等建单、乐观锁、越权拒绝、附件可见性和审计日志有效。
- 工单历史、协同记录、反馈、申诉、通知、回访任务、AI 建议均可落库。
- Backend、Action、TypeScript、前端构建和基础迁移已有可用基础。

### 当前缺口

当前约为“可演示 MVP 82% / 生产就绪 55%”。主要缺口是可靠交付，而不是页面数量：

- 默认 Vitest 失败，Rasa 离线测试和 Playwright 缺少有效通过证据；
- Rasa tracker 为内存存储，重启后会话丢失；
- Alembic 与 ORM 元数据漂移；
- 临期提醒依赖打开通知页触发，没有可靠调度和重试；
- PostgreSQL 备份不包含 MinIO 附件；
- 附件恶意文件扫描未启用；
- 部门同步按钮、裸工单号查询、AI 历史建议加载存在断链；
- OIDC、短信、地图、目录、政务工单等未接入；
- 当前 AI 是规则建议，不是真实大模型。

## 2. 新版产品定位与范围

倾听助手定位为面向政务诉求、企业客服和公共事务场景的“四角色智能工单协同系统”。

核心价值：

1. 对话式诉求登记；
2. 可信状态机和权限控制；
3. 跨角色办理协同；
4. 附件、通知、回访、申诉和审计证据链；
5. AI 辅助研判但不替代人工决策；
6. 可重复部署、测试、备份和恢复。

### 最终交付标准

- 一条命令启动完整演示环境；
- 四角色使用固定演示账号完成主闭环；
- 默认测试、迁移检查和 E2E 全部通过；
- 服务重启后业务数据、附件和 Rasa 会话不丢失；
- PostgreSQL 与 MinIO 可联合备份、恢复和校验；
- 可通过域名和 HTTPS 部署到单台服务器；
- README、架构、脚本、测试报告和代码保持一致。

### 本轮不做

- Kubernetes、多机高可用、跨机房灾备；
- 真实政府目录和全部第三方渠道；
- 自动行政决策、自动拒绝、自动派发、自动办结；
- 自动电话外呼；
- 复杂 BPMN、多租户 SaaS、原生 App 或小程序；
- 为丰富技术栈而引入 Qdrant、Elasticsearch、Kafka 等重型组件。

## 3. 四端完整功能

### 市民端

- 对话或表单创建投诉、建议、咨询和求助；
- 补充信息、上传附件、查询本人诉求；
- 查看状态、办理轨迹、通知和回访结果；
- 催办、满意/不满意反馈、申诉；
- 查看面向市民的 AI 摘要和材料完整性提示。

### 坐席端

- 待受理、待派发、办理中和待协调队列；
- 受理、拒绝、分类、优先级确认和部门派发；
- 请求补充材料、协调协办/复核、催办；
- 查看 SLA、临期和超期风险；
- 管理回访任务和人工电话记录；
- 复核 AI 分类、部门和风险建议。

### 部门端

- 查看本部门主办、协办和复核任务；
- 开始办理、添加记录和上传附件；
- 退回、转派、协办、复核和提交结果；
- SLA 暂停、恢复和超期说明；
- 查看 AI 摘要、草稿和风险提示；
- 不显示无权限或未配置的外部同步操作。

### 管理员端

- 全局工单、SLA、部门负载和申诉统计；
- 用户、部门、分类和基础 SLA 策略管理；
- 工单协调、申诉审核和异常干预；
- 审计日志、敏感读取和 AI 建议复核；
- 系统健康、版本和集成状态只读页面；
- 不建设可在线修改秘密和令牌的平台配置中心。

## 4. 核心业务流程与状态机

```text
pending
  ├─ accept → accepted
  └─ reject → rejected

accepted ─ assign → assigned
assigned ─ process → processing

processing
  ├─ note → processing
  ├─ return/transfer → assigned 或 processing
  └─ resolve → resolved

resolved
  ├─ citizen satisfied → closed
  ├─ citizen dissatisfied → processing
  └─ approved appeal → processing
```

协同任务保留 `primary`、`support`、`review` 三类：

```text
pending → processing → submitted
   ├─ returned
   ├─ transferred
   └─ cancelled
```

规则：

- 所有状态变化使用命名动作接口，前端不得直接写任意状态；
- 主工单和协同任务均保留独立版本号；
- 申诉审核通过后回到 `processing`；
- 办结必须记录 closure reason、操作者、时间和版本。

## 5. AI 边界与升级

### 必须保持

- AI 只提供建议，不修改工单、状态、权限或责任部门；
- 所有输出保持 `advisory_only=true`；
- 记录 provider、model、prompt_version、输入摘要、输出、耗时和人工复核；
- 模型不可用时自动降级为现有规则模式；
- AI 草稿不能直接成为正式行政文书。

### 推荐增加的真实 AI 能力

仅接入一个可配置的 OpenAI 兼容模型：

1. 分类、优先级和责任部门建议；
2. 工单摘要、完整性检查和答复草稿；
3. 风险提示及相似历史工单解释。

相似检索优先使用 PostgreSQL 全文检索和 `pg_trgm`，不新增独立向量数据库。

## 6. 页面、接口和数据库调整

### 页面

- 修复部门端无权限同步按钮；
- Chat 支持直接识别标准工单号；
- Intelligence 页面加载历史建议及复核状态；
- 坐席派发完整加载部门和人员；
- 统一四角色空状态、错误、加载和 409 冲突提示；
- 管理端新增系统状态页；
- 清理 NaN 样式、弃用警告和过大前端包。

### 接口

保留命名业务动作接口，逐步废弃：

- 通用 `PATCH /tickets/{id}/status`；
- 与详情响应重复且无调用方的接口；
- 含义重复的 health alias。

补齐：

- 401、403、409、413、429、500 OpenAPI 契约；
- AI 历史建议查询和分页；
- 系统版本、迁移版本、任务和依赖状态；
- 后台任务和通知 outbox 的只读管理及重试接口；
- 统一幂等键、版本冲突和错误码。

### 数据库

保留现有业务表，不推倒重建：

- 修正 ORM、迁移、索引、Identity 和约束漂移；
- 确保 `alembic check` clean；
- 新增 `sla_policies`；
- 新增 `notification_outbox`，记录状态、重试次数和下次执行时间；
- 为 AI 建议补充模型、提示词版本、耗时和错误字段；
- Rasa tracker 使用 PostgreSQL 独立 schema 或独立数据库；
- 补充常用状态、部门、时间和审计查询索引。

不增加通用 `ticket_processing_records`，继续使用 `ticket_status_history` 和 `work_order_history`。

## 7. 工程化与部署

### 服务结构

在现有七服务基础上增加：

- `worker`：临期扫描、通知 outbox、失败重试和周期任务；
- `clamav`：生产配置下的附件恶意文件扫描。

维护三类配置：

- 开发环境；
- 固定 Seed 的演示环境；
- 强秘密、HTTPS、扫描、持久卷和备份的生产等价环境。

### 安全

- 生产模式拒绝弱 JWT、默认 MinIO 密码和宽泛 CORS；
- HTTPS 反向代理只暴露必要端口；
- 附件执行大小、扩展名、MIME、哈希和 ClamAV 扫描；
- 修复仅依赖 `Content-Length` 的请求体限制；
- 登录限流改为共享存储或数据库实现；
- CI 依赖漏洞检查改为阻断门禁。

### 可观测性

- 保留 JSON 日志和全链路 `request_id`；
- Docker 日志开启大小和文件数量轮转；
- Backend、Action、Rasa、worker 输出服务名、请求 ID、工单 ID 和耗时；
- 提供基础 metrics endpoint；
- 健康检查区分 live、ready 和 degraded。

### 备份恢复

统一备份包：

```text
backup/
├─ postgres.dump
├─ minio/
├─ manifest.json
└─ checksums.sha256
```

恢复时必须校验：

- PostgreSQL 数据；
- MinIO 对象；
- 附件元数据与对象一一对应；
- 对象大小和 SHA-256；
- 缺失、孤立和损坏对象清单。

## 8. 分阶段开发计划

### 阶段一：稳定基线与质量门禁（P0）

- 定位 Rasa 离线测试卡死及 Docker 异常；
- 修复默认 Vitest 超时和异步 teardown；
- 建立无端口冲突的 Playwright 环境；
- 修复 Alembic/ORM 漂移；
- 更新过期 demo、E2E 和迁移脚本；
- CI 中所有测试和安全检查改为阻断门禁。

验收：

- 默认命令连续两次通过；
- `alembic check` clean；
- 空库安装和已有库升级均通过；
- 七服务启动、停止和重启稳定；
- 不通过放大 timeout 掩盖问题。

### 阶段二：业务闭环与体验补齐（P1）

- 修复部门按钮、裸工单号查询和 AI 历史建议；
- 完善派发和四角色错误交互；
- 接入 PostgreSQL Rasa tracker；
- 明确匿名会话与登录市民工单的绑定；
- 增加 worker、SLA 定时扫描和通知 outbox；
- 增加简化 SLA 策略管理；
- 完成四角色主流程和越权 E2E。

验收：

- Rasa 重启后会话可继续；
- 无人访问通知页也会生成临期提醒；
- 失败任务可重试且不重复通知；
- 页面不存在点击必然 403 的操作。

### 阶段三：安全部署与恢复交付（P0/P1）

- 增加生产等价 Compose 和 HTTPS；
- 强秘密、CORS、端口和 config guard；
- 接入 ClamAV；
- 修复请求体限制和共享限流；
- 完成 PostgreSQL + MinIO 联合备份恢复；
- 增加日志轮转、指标和任务状态；
- 完成部署、升级、回滚和恢复文档。

验收：

- 弱秘密或未启用扫描时生产配置拒绝启动；
- 全新服务器可按文档部署；
- 隔离环境恢复演练通过；
- 恢复后工单、附件、审计和 Rasa 会话完整。

### 阶段四：AI 与求职展示（P2，推荐）

- 接入 OpenAI 兼容模型并保留规则降级；
- 实现分类建议、摘要草稿和风险提示；
- 完整记录模型、提示词和人工复核；
- 准备三条固定演示剧本；
- 更新 README、架构图、接口、测试报告和简历描述；
- 隔离无引用的旧 Chatroom、ServiceNow local 和陈旧资产。

验收：

- 无模型密钥时仍可完整演示；
- AI 不改变业务状态；
- 输出可追溯、可复核、可回放；
- 10 分钟内稳定演示主闭环和申诉流程。

## 9. 最终验收标准

### 功能

- 四角色只能访问授权数据；
- 对话和表单均可创建工单；
- 受理、派发、办理、协同、反馈、申诉和办结完整；
- 状态机、幂等和乐观锁有效；
- 附件权限、扫描、下载和删除正确；
- 通知、回访、SLA 和申诉记录完整；
- AI 只建议且可复核；
- 所有关键动作可由审计日志追踪。

### 测试

默认命令全部通过：

- Backend pytest、Ruff、编译检查；
- Action 单元和 HTTP gateway 测试；
- Rasa validate、Core、NLU 和 overlap；
- Frontend TypeScript、Vitest、构建；
- Playwright 三浏览器主流程；
- Alembic check、空库迁移、升级迁移和 Seed 幂等；
- 备份恢复、附件一致性和生产配置安全测试。

### 部署

- 全新环境一键启动且服务 healthy；
- 提供固定演示账号和可重复 Seed；
- 支持 HTTPS、持久卷和生产秘密；
- 单服务重启不丢失数据和会话；
- 备份包可在隔离环境恢复；
- 连续 30 分钟演示和基础并发检查无持续 5xx、数据错乱或重复建单。

- `README.md`

- 开发、演示、生产等价 Compose
- `.env.example`

- API 文档和状态机图
- 固定演示数据和三条演示剧本
- 当前真实测试报告和已知边界
- 简历项目描述和面试技术点

## 10. 实施原则

- 不推倒现有架构，优先修复已验证有效的主链；
- 先完成阶段一，再新增功能；
- 每阶段必须有代码、运行、测试和文档证据；
- 不把 Mock、未配置适配器或规则建议包装成真实系统；
- 不为丰富技术栈增加不必要基础设施；
- README、简历和演示表述必须与最终运行结果一致。
