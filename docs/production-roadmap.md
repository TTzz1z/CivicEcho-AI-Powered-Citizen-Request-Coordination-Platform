# 生产化演进路线图

## 当前定位

V1.0 是单机 Docker Compose 的求职展示与工程基线，不等同于已完成生产上线。路线图遵循“先补运行保障与安全，再做容量与组织集成，最后按证据演进架构”，不在当前版本引入 Kubernetes、ELK、Prometheus 或微服务拆分。

## 阶段 0：上线前必做（P0）

| 方向 | 建议 | 验收证据 |
|---|---|---|
| 网络边界 | TLS、可信反向代理/WAF、Host 白名单、只暴露 frontend | 外部扫描与代理配置审计 |
| 秘密 | 使用云或部署平台 Secret Manager，轮换 JWT/服务/数据库密钥 | 仓库与镜像无秘密，轮换演练 |
| 身份 | 接入 OIDC/统一身份，禁用演示账号与公开 Seed | SSO、登出、禁用、审计测试 |
| 数据库 | 托管 PostgreSQL、HA、连接池、只允许私网访问 | 故障切换与连接耗尽测试 |
| 备份 | 加密异地备份，定义 RPO/RTO，季度恢复演练 | 可复核的恢复报告 |
| 安全升级 | 固定月度依赖扫描与兼容升级窗口 | 无未接受高危项，例外有到期日 |
| 发布 | 制品签名、SBOM、不可变镜像、迁移与回滚 Runbook | 灰度/回滚演练 |

## 阶段 1：稳定运行（P1）

- 把 stdout 日志接入组织现有集中日志平台，建立 5xx、登录爆破、队列积压、数据库连接和依赖失败告警。
- 定义 SLI/SLO：API 可用率、P95 延迟、建单成功率、Action 超时率、工单积压和平均办理时长。
- 将进程内登录限流迁移到 Redis 或网关级限流；Rasa tracker 也改为共享持久化，支持多实例。
- 引入任务队列处理通知、重试和非关键副作用，保留 Backend 事务为真相源。
- 做容量与故障测试：数据库断连、Action/Rasa 超时、磁盘满、慢查询和依赖降级。
- 规范数据保留、脱敏、删除、更正和审计访问审批，匹配真实政务合规要求。

## 阶段 2：业务接入（P1/P2）

- 通过防腐层接真实政务工单或 ServiceNow，先做双写比对/只读同步，再逐步切换。
- 建立组织、区域、事项分类、SLA、催办、通知和附件的正式领域模型。
- 附件进入对象存储并做类型、大小、恶意文件扫描；不直接写应用容器或数据库大字段。
- 基于真实标注集重新评估中文 NLU，建立意图 F1、实体准确率、fallback 率和漂移监控。
- Rasa 主版本迁移独立立项，保留模型与故事基准，不能与业务大改同批发布。

## 阶段 3：按证据扩展架构（P2）

只有出现明确瓶颈时才考虑 Kubernetes 或拆分服务：

- Rasa 与 Backend 的扩缩容曲线明显不同；
- 独立团队拥有清晰数据边界和发布节奏；
- 单体数据库/部署已经成为可量化瓶颈；
- 组织已有成熟的平台、可观测性和故障处理能力。

可能的边界是“对话编排”“工单核心”“通知集成”“分析读模型”，但拆分前先用模块边界、异步事件与读副本验证收益。

## 明确不建议立即做

- 为了简历术语直接拆微服务；
- 在没有数据治理前加入大模型/RAG；
- 把审计日志当作通用业务事件总线；
- 用自动 downgrade 代替数据库向前修复；
- 在未建立 SLO 和容量基线前引入复杂编排平台。
